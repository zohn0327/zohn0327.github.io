Web安全

WEB安全基础

常见请求方法及功能

GET : 要求从请求的URL获取资源
POST : 要求服务器接受附带的正文信息。与GET请求相同，但随请求一起发送了额外的信息
HEAD : 仅要求GET将返回的任何标头部分。就像GET，但没有正文
TRACE : 要求环回请求消息，以进行测试或故障排除。 备注：回显服务器收到的请求，主要用于测试或诊断。一般禁用，防止被恶意攻击或盗取信息。
PUT : 表示将随附的信息(正文)放在请求的URL上。 备注：抓包后可修改上传文件并指定任意web站点下的路径进行上传
DELETE : 表示删除请求URL处的资源 备注：会删除指定url的资源
OPTIONS : 请求web服务器告知其支持的各种功能 备注：获取http服务器支持的http请求方法，允许客户端查看服务器的性能
MOVE
COPY
GET与POST区别？
GET以明文方式通过RUI提交数据，数据在URL中可以看到，提交数据不超过2K，安全性较低但传输效率较高，适合查询、搜索类安全性不高的操作；
POST将用户提交的数据封装在Header内，适用于提交数据量较大，提交数据可达2M，适用注册、修改等安全性较高的用户信息

常见状态码及理解

1xx：指示信息–表示请求已接收，继续处理。
2xx：成功–表示请求已被成功接收、理解、接受。
3xx：重定向–要完成请求必须进行更进一步的操作。
4xx：客户端错误–请求有语法错误或请求无法实现。
5xx：服务器端错误–服务器未能实现合法的请求。常见状态代码、状态描述的说明如下。
200 OK：客户端请求成功。
400 Bad Request：客户端请求有语法错误，不能被服务器所理解。
401 Unauthorized：请求未经授权，这个状态代码必须和WWW-Authenticate报头域一起使用。
403 Forbidden：服务器收到请求，但是拒绝提供服务。
404 Not Found：请求资源不存在，举个例子：输入了错误的URL。
500 Internal Server Error：服务器发生不可预期的错误。
503 Server Unavailable：服务器当前不能处理客户端的请求，一段时间后可能恢复正常，举个例子：HTTP/1.1 200 OK（CRLF）。
部分恶意内容被WAF拦截后，响应为HTTP/1.1 406 Not Acceptable

常见web漏洞原理及修复加固建议：
sql注入：后台未对用户输入的数据进行严格校验，且输入的数据被带入到数据库并成功作为数据库执行语句进行执行；
联合查询注入：
报错注入：
布尔盲注：
时间盲注：
对查询注入：
宽字节注入：
二次注入：
防御方法：
代码层采用预编译，强制使用参数化语句，并对用户输入进行严格校验；
数据库层针对数据库敏感行为进行监控，禁止使用特权账户；
设备层：web前部署WAF
文件上传：未对用户上传的文件类型做严格校验，使用户能够上传webshell到web服务器
防御方法：
隐藏上传路径；
上传文件使用随机字符+时间戳重命名；
将上传文件夹写死不执行动态脚本文件；
设置单独的文件服务器；
检查文件上传路径，避免00阶段、IIS6的文件夹解析漏洞、目录遍历
文件扩展名检测，避免服务器以非图片的文件格式解析文件，建议使用白名单的方式
后端验证文件MIME，防止Content-Type前端绕过
后端检查文件头内容进行文件内容检测16进制，避免图片马
XSS跨站脚本攻击：DOM型、反射型、存储型
危害：
修复：
针对用户输入进行严格校验，过滤js标签和字符，针对页面的输出环境进行编码；
Cookie设置Http-only禁止客户端访问cookie
文件包含漏洞：include、include_once、require、require_once
危害：
修复建议：
严格判断包含中的参数是否外部可控、包含路径限制只能在某一文件夹内，禁止跳转目录、包含文件验证都是白名单中的一员、尽量不使用动态包含并将需要包含的页面固定写好；
越权漏洞：水平越权和垂直越权
水平越权危害：获取到具备同级权限的其他用户的敏感信息或非法操作权限；
垂直越权危害：获取到更高级别用户的敏感信息或非法操作权限；
修复建议：
执行操作前必须验证用户身份，检查用户是否具备操作数据的权限
前后端同时对用户输入信息进行校验，双重验证机制；
从用户的加密认证cookie中获取当前用户id，防止攻击者对其修改；
直接对象引用的加密资源id，防止攻击者枚举ID,敏感字符特殊化处理；

逻辑漏洞篇：
除了涉及用户管理类的越权漏洞外，逻辑漏洞较多的存在登录、注册、修改密码模块；
1、验证码前端可控，形同虚设；
2、验证码未做销毁处理，可重复使用，从而可爆破或注册、密码找回功能进行短信轰炸；
3、验证码逻辑简单，可进行空字符串绕过或猜解；
4、验证码或重置密码的链接直接暴漏在响应包中
5、提示信息不严谨，如密码错误时提示密码错误，账号不存在时提示账号不存在，可遍历已注册账户账号；
加固方式：
验证码进行后端校验；
验证码进行验证一次后做失效处理；
输入多次错误后进行账户锁定；
使用强密码防止爆破；
提示信息要严谨，验证码的发送情况、注册情况只有服务器和用户本人能够确定是否正在进行操作；
